비모 유니버스 — 인증 / SSO / ERD

하나의 Supabase 프로젝트에 auth.users 단일 소스를 두고, app_access 테이블로 앱별 권한을 관리. 비모 ERP · Partner ERP · Vibox 세 앱은 같은 JWT로 SSO가 가능하며, 이후 다른 서비스(영상 검수, SEON Hub, Hype Kit 등)를 동일 계정으로 손쉽게 추가할 수 있는 확장형 구조.

4 · SSO 로그인 흐름

Supabase는 JWT 기반이라 같은 루트 도메인이면 쿠키 자동 공유로 SSO가 되고, 서로 다른 도메인이면 redirect-based 핸드오프로 구현 가능. 

sequenceDiagram
    autonumber
    participant U as 사용자
    participant V as Vibox (vibox.cloud)
    participant S as Supabase Auth
    participant P as Partner ERP (partner.vi-mo.kr)

    U->>V: Vibox 방문 (로그인 안된 상태)
    V->>S: 로그인 요청 (email/password)
    S-->>V: JWT access_token + refresh_token
    V->>V: 쿠키 저장 (.vi-mo.kr 도메인)
    Note over V: SSO 쿠키 세팅

    U->>P: Partner ERP 접속
    P->>P: 쿠키에서 JWT 확인
    alt 쿠키 공유 가능 (같은 루트 도메인)
        P->>S: JWT 검증
        S-->>P: user_id + app_access 조회
        P-->>U: 자동 로그인 완료 ✓
    else 다른 도메인 (핸드오프 필요)
        P->>V: SSO 엔드포인트로 redirect
        V->>P: 일회용 토큰 + return_url
        P->>S: 토큰 검증 → JWT 교환
        P-->>U: 로그인 완료 ✓
    end
1
동일 루트 도메인 구성 (가장 권장)
vi-mo.kr 아래 서브도메인으로 배포 → erp.vi-mo.kr, partner.vi-mo.kr, vibox.vi-mo.kr. 쿠키 도메인을 .vi-mo.kr로 설정하면 세 앱이 자동으로 JWT 공유. 별도 코드 필요 없음.
2
다른 도메인이라면 (Vibox → vibox.cloud)
redirect-based SSO: Vibox에서 로그인 후, 다른 앱 접속 시 "Sign in with 비모" 버튼 → Supabase Auth endpoint로 redirect → OAuth 2.0 스타일 토큰 핸드오프. Supabase는 이걸 네이티브 지원.
3
앱 로드 시 app_access 체크
로그인 후 각 앱은 app_access WHERE user_id = auth.uid() AND app_code = 'vimo_erp' AND status = 'active' 쿼리로 접근 권한 확인. 없으면 "권한이 없습니다" 안내.
4
RLS로 데이터 격리
각 앱의 테이블 RLS 정책에 auth.uid() IN (SELECT user_id FROM app_access WHERE app_code = 'X') 조건 추가 → 해당 앱 권한 있는 유저만 데이터 접근 가능.

4.5 · 팀 ERP ↔ 파트너 ERP 상호 배제 제약

한 유저는 비모 팀 ERP 또는 파트너 ERP 둘 중 하나만 접근 가능해야 함 (Vibox는 양쪽 모두 허용). 실수로 동시 권한 부여되는 걸 DB 레벨에서 원천 차단.

DB 제약 · 부분 유니크 인덱스

CREATE UNIQUE INDEX app_access_erp_exclusive
ON public.app_access (user_id)
WHERE app_code IN ('vimo_erp', 'partner_erp');

→ 한 user_id는 vimo_erp와 partner_erp 중 최대 1개 row만 가질 수 있음. 두 번째 INSERT 시도 시 Postgres가 reject.

추가 안전장치 · user_type 컬럼

profiles.user_type enum('staff', 'partner', 'external') 추가 후 가입 시 확정, 이후 변경 불가. app_access INSERT 트리거에서 user_type과 app_code 매칭 검증. 

-- staff는 vimo_erp만, partner는 partner_erp만 부여 가능
CREATE OR REPLACE FUNCTION check_app_access_type()
RETURNS trigger AS $$
BEGIN
  IF NEW.app_code = 'vimo_erp' AND
     (SELECT user_type FROM profiles WHERE id = NEW.user_id) != 'staff'
  THEN
    RAISE EXCEPTION 'vimo_erp는 staff 유저만 접근 가능';
  END IF;
  IF NEW.app_code = 'partner_erp' AND
     (SELECT user_type FROM profiles WHERE id = NEW.user_id) != 'partner'
  THEN
    RAISE EXCEPTION 'partner_erp는 partner 유저만 접근 가능';
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER enforce_app_access_type
BEFORE INSERT OR UPDATE ON app_access
FOR EACH ROW EXECUTE FUNCTION check_app_access_type();

1
비모 팀 측 등록 화면 (관리자 전용)
비모 ERP 안에 "파트너 초대" 페이지 → 이름/유형/기수 입력 → "초대 링크 생성" 클릭 → profiles + partners(status=pending) + partner_invites(token) 동시 생성 → 생성된 URL과 QR을 복사 가능하게 표시.
2
파트너별 전용 URL
https://partner.vi-mo.kr/onboard?token=xK9aB2cD... (UUIDv4 또는 crypto random 32chars). URL에 이름을 노출 안 해도 토큰이 profile과 연결돼있어서 열면 자동으로 "안녕하세요, 서유진님" 표시.
3
보안 규칙
7일 만료 만료 시 status=expired · 일회용 사용 완료 시 재접속 불가 · 재발급 필요 시 관리자가 기존 토큰 revoke하고 새로 발급 · 이메일 검증 원하면 초대 시 이메일 고정해서 해당 이메일로만 가입 허용
4
가입 완료 시 자동 처리
auth.users 생성 → profile_id 매핑 → app_access에 partner_erp + vibox 동시 부여 → partners.status = 'active' → invite.used_at = now() → 첫 로그인 → 온보딩.

1
권한 레벨
view 미리보기만 · download 다운로드 허용 · upload 외부인이 해당 폴더에 파일 업로드 (Dropbox File Request 스타일) — 클라이언트 피드백·외주 소스 수집에 유용
2
RLS (Row-Level Security) 처리
Vibox 파일 테이블 RLS에 anonymous 접근 조건 추가: auth.uid() = owner_id OR EXISTS (SELECT 1 FROM vibox_shares WHERE resource_id = files.id AND token = current_setting('request.share_token', true) AND status = 'active') → 로그인 없이 유효한 토큰만으로 특정 row만 접근 허용
3
감사 추적 (visitor_log)
누가(이메일), 언제, 어디서(IP), 뭘(view/download) 했는지 기록. 비모 팀은 대시보드에서 "이 파일 지금까지 15명 열람 · 3명 다운로드" 확인 가능. 의심스러우면 즉시 revoke.
4
유저 타입 정리
user_type enum이 4종류로 확장 → staff (비모 팀) · partner (파트너) · external_member (비모 계정 있는 외부 협력사 · 선택) · 비회원은 user_type 없음 (auth.users 자체가 없고 visitor_log에만 존재)

5 · 향후 서비스 확장 (비모 유니버스)

이 구조의 핵심 장점 — 새 서비스 추가 시 단 3단계만 필요:

① apps 에 새 앱 등록
INSERT INTO apps (code, name, domain) VALUES ('video_qa', '영상 검수 AI', 'qa.vi-mo.kr')
② 대상 유저에게 app_access 부여
INSERT INTO app_access (user_id, app_code, role) VALUES (...) — 비모 팀 / 파트너 / 특정 유저 누구든 지정 가능
③ 새 앱의 테이블에 RLS 적용
USING (auth.uid() IN (SELECT user_id FROM app_access WHERE app_code = 'video_qa'))

확장 예시 영상 검수 AI → video_qa 앱 등록 → 비모 팀에게만 접근 부여 → Dropbox 영상 검수 프로덕트로 바로 연결

외부 확장 Hype Kit / SEON Hub 같은 외부 브랜드 서비스도 apps에 등록만 하면 비모 계정 단일 로그인으로 이용 가능 — OAuth 스타일로 외부 제공도 가능

6 · 현 상태에서의 마이그레이션

현재 (As-is)

비모 ERP — Supabase 프로젝트 A (auth.users + 직접 profile 관리) · Vibox — 별도 SQLite 기반 (분리) · Partner ERP — 아직 없음

목표 (To-be)

Supabase 프로젝트 A 단일 → profiles, app_access, apps 테이블 추가 → 비모 ERP는 그대로 두고 app_access로 접근 체크 추가 → Partner ERP는 같은 프로젝트로 새로 개발 → Vibox는 Supabase로 마이그레이션 (기존 유저는 이메일 매칭으로 연결)

단계별

A
공용 스키마 추가
현재 Supabase에 profiles, app_access, apps 테이블 생성. 기존 비모 ERP 로그인은 app_access에 vimo_erp 항목 있는지로 판정.
B
Partner ERP 런칭
같은 Supabase에 partner_erp 전용 테이블(내 할일·정산·거래처) 추가. 파트너는 app_access에 partner_erp 등록해서 로그인 허용.
C
Vibox 통합
Vibox를 Supabase Auth로 이전. 기존 Vibox 유저는 이메일 기준 매핑 → 초대 메일 발송 → 비번 재설정으로 비모 계정과 병합. 이후 SSO 쿠키 공유로 심리스 전환.
